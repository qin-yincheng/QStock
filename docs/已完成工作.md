- 产出物：明确采用方案A

### Step 1.2 补充 — Bug 定位

- 问题根因：czsc `ts_connector.py` 的 `get_raw_bars()` 函数中：
  1. `Freq.F60` 的枚举值是 `"60分钟"`
  2. 代码将其替换为 `"60min"` 去调用 tushare API（成功）
  3. 但随后用 `Freq("60min")` 构造枚举对象时报错（因为枚举值是 `"60分钟"` 而非 `"60min"`）
- 解决方案：在 `core/data_provider.py` 中封装修复逻辑，绕过此 bug

### Step 1.3 + 1.5 — 数据通路修复与统一接口 ✅

- 发现 czsc 库两个 bug：
  1. **freq 枚举 bug**：`get_raw_bars` 将 `"60分钟"` 替换为 `"60min"` 后尝试 `Freq("60min")`，但枚举值是 `"60分钟"`
  2. **列名不匹配 bug**：`pro_bar_minutes` 标准化输出为 `symbol/dt` 列，但 `format_kline` 期望 `ts_code/trade_time` 列
  3. **附加问题**：czsc 的 `Freq` 来自 Rust 扩展 `rs_czsc`，不可哈希，不能用作 dict key
- 创建 `core/data_provider.py`：统一数据入口，绕过以上所有问题
  - 分钟数据：直接调用 `pro_bar_minutes` + 手动构造 RawBar（不走 `format_kline`）
  - 日线数据：调用 `ts.pro_bar` + `format_kline`（日线正常）
  - 使用 `str(freq)` 字符串匹配代替字典映射
- 创建 `tests/test_czsc_60min.py`：验证脚本
- 测试结果：
  - ✅ 60分钟：516 条 RawBar，时间 2025-09-01 ~ 2026-03-18
  - ✅ 日线：48 条 RawBar
  - ✅ 数据质量检查通过
- 产出物：完整的数据获取通路，第一阶段全部完成
